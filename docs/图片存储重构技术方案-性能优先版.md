# 图片存储重构技术方案（最终版，性能优先）

> 状态：定稿
> 日期：2026-06-08
> 前提：项目无线上历史数据，本方案不考虑旧字段兼容、灰度迁移和数据回填。
> 目标：在保持实现轻量的前提下，补齐权限边界、失败补偿和删除策略。

---

## 一、设计原则

1. **性能优先**：上传主链路不引入重型校验，不做每次上传查 `pet_spaces`、不做上传成功后的额外同步落库。
2. **安全足够**：上传权限只允许写入服务端签发过的 key 范围，不能靠前端传 `petSpaceId` 直接决定路径。
3. **失败可收敛**：不追求强事务，但要让孤儿文件可被快速、低成本回收。
4. **无兼容包袱**：直接使用最终 schema、最终接口、最终上传链路，不保留旧字段、不保留旧分支。
5. **实现克制**：当前只接七牛，不提前实现多 provider 运行时分发。

---

## 二、最终结论

保留 `Ref = { storage, bucket, key }` 作为唯一入库格式，但把上传授权改成两层短期凭证：

1. `sessionGrant`
   - 登录态内短期有效，承载 `openid + uid`。
   - 用于头像等“只和当前用户相关”的上传。
   - 作用是**去掉每次上传时的 `openid -> uid` 查库**。
2. `petUploadGrant`
   - 进入“创建宠物 / 编辑宠物 / 写记忆”页面时，由服务端一次性签发。
   - 承载 `uid + petSpaceId + scope + exp`。
   - 用于宠物封面、相册、记忆图片上传。
   - 作用是**去掉每次上传时的宠物归属查库**，同时防止用户伪造别人的 `petSpaceId`。

`getQiniuUploadToken` 只做以下事情：

1. 校验微信登录态存在。
2. 校验 `sessionGrant` 签名、过期时间、`openid` 一致性。
3. 针对宠物图片，校验 `petUploadGrant` 签名、过期时间、`uid` 一致性、scope 是否匹配。
4. 生成最终 key。
5. 按精确 `bucket:key` 签发七牛上传 token。

整个过程不查 `users`、不查 `pet_spaces`，因此热路径非常轻。

## 三、数据结构

### 3.1 Ref

```js
{
  storage: 'qiniu',
  bucket: 'cl8023',
  key: 'project/star-pet/uploads/users/{uid}/pet-spaces/{petSpaceId}/memories/1733xxxxxx-ab12cd34.jpg'
}
```

约束：

1. DB 只存 `storage / bucket / key`，不存 url。
2. `storage` 当前固定为 `qiniu`。
3. `bucket` 当前固定为 `cl8023`。

### 3.2 SessionGrant

建议使用紧凑 JSON + HMAC-SHA256 签名，Base64URL 编码：

```js
{
  v: 1,
  openid: 'o_xxx',
  uid: 'Kx3pQ9aB2mNs',
  exp: 1760000000000
}
```

规则：

1. 有效期 24 小时。
2. 由 `login` 返回给客户端。
3. 客户端只透传，不信任客户端改写内容。

### 3.3 PetUploadGrant

```js
{
  v: 1,
  uid: 'Kx3pQ9aB2mNs',
  petSpaceId: '64f4...',
  scope: ['petCover', 'petAlbum', 'memory'],
  exp: 1760000000000
}
```

规则：

1. 有效期 20 分钟。
2. 创建宠物页由 `reservePetSpaceId` 返回。
3. 编辑宠物/写记忆页由轻量云函数 `getPetUploadGrant` 返回。
4. 只承载上传授权，不承载业务写权限。

## 四、关键优化点

### 4.1 不在上传热路径查库

原方案的两次潜在查库：

1. `openid -> uid`
2. `petSpaceId -> owner`

都挪到“页面初始化”或“登录态初始化”时做一次，然后通过短期签名凭证复用。

收益：

1. 单次上传签 token 只做本地验签和字符串拼接。
2. 多图场景下收益明显，尤其是记忆页连续上传 3 到 9 张图时。
3. 云函数耗时和数据库压力更稳定。

### 4.2 不做 provider 运行时抽象

只保留：

1. `utils/storage/index.js`
2. `utils/storage/qiniu.js`
3. 云函数共享 `storage.js`

不做 `switch(provider)` 的运行时分发，避免多一层抽象和测试面。当前实现明确是七牛专用实现。

### 4.3 URL 继续不入库

读取云函数统一使用 `buildUrl(ref)` 拼接返回，原因不变：

1. 零外部 API 调用。
2. CDN 域名切换只改配置。
3. 读路径性能足够高，字符串拼接成本可忽略。

## 五、上传路径规范

统一 key：

```txt
project/star-pet/uploads/users/{uid}/{subPath}/{ts}-{rand}.{ext}
```

其中：

1. `{ts}` 为毫秒时间戳。
2. `{rand}` 为 8 字节 hex。
3. `{ext}` 从客户端归一化结果决定。
   - 如果裁剪链路统一输出 JPEG，则固定 `jpg`。
   - 如果保留原格式，则必须从 MIME/文件名白名单推导 `jpg|jpeg|png|webp`。

`subPath`：

| type | subPath |
|---|---|
| `avatar` | `avatars` |
| `petCover` | `pet-spaces/{petSpaceId}/covers` |
| `petAlbum` | `pet-spaces/{petSpaceId}/albums` |
| `memory` | `pet-spaces/{petSpaceId}/memories` |

约束：

1. `petSpaceId` 只从服务端签发的 `petUploadGrant` 取，不直接信前端入参。
2. 不再允许“只传 petSpaceId 就给 token”的模式。

## 六、接口设计

### 6.1 `login`

新增返回：

```js
{
  user: { ... },
  sessionGrant: 'base64url(payload).base64url(signature)'
}
```

职责：

1. 新用户创建时生成 `uid`。
2. 返回 `sessionGrant`。

### 6.2 `reservePetSpaceId`

返回：

```js
{
  petSpaceId: '64f4...',
  petUploadGrant: '...'
}
```

职责：

1. 预分配宠物 `_id`。
2. 直接签发绑定该 `petSpaceId` 的 `petUploadGrant`。

### 6.3 `getPetUploadGrant`

用于编辑宠物、写记忆页的轻量授权刷新。

入参：

```js
{ petSpaceId }
```

返回：

```js
{ petUploadGrant }
```

服务端逻辑：

1. 查一次 `pet_spaces`，确认当前用户拥有该宠物。
2. 签发 20 分钟有效的 `petUploadGrant`。

说明：这个查库发生在“进入编辑页”时，一页一次，不在每张图片上传时发生。

### 6.4 `getQiniuUploadToken`

入参：

```js
{
  type: 'avatar' | 'petCover' | 'petAlbum' | 'memory',
  sessionGrant: '...',
  petUploadGrant: '...', // avatar 可不传
  ext: 'jpg'
}
```

返回：

```js
{
  ok: true,
  uploadToken: '...',
  key: 'project/star-pet/uploads/users/...jpg',
  ref: {
    storage: 'qiniu',
    bucket: 'cl8023',
    key: 'project/star-pet/uploads/users/...jpg'
  },
  url: 'https://qiniu.cdn.cl8023.com/project/star-pet/uploads/users/...jpg'
}
```

服务端逻辑：

1. 从 `cloud.getWXContext()` 取 `openid`。
2. 验 `sessionGrant.openid === openid`。
3. 从 `sessionGrant` 取 `uid`。
4. 如果 `type !== avatar`，验 `petUploadGrant.uid === uid`，且 `scope` 包含当前 type。
5. 用 grant 里的 `petSpaceId` 生成最终 key。
6. 以 `bucket:key` 签七牛 token。

说明：返回 `ref + url`，前端上传后不用自己再拼。

### 6.5 `cleanupRefs`

入参：

```js
{
  sessionGrant: '...',
  refs: Ref[]
}
```

返回：

```js
{
  ok: true,
  deleted: {
    successCount: 2,
    failCount: 0,
    failedKeys: []
  }
}
```

服务端逻辑：

1. 校验微信登录态。
2. 校验 `sessionGrant`，取出 `uid`。
3. 只允许删除 `project/star-pet/uploads/users/{uid}/` 前缀下的对象。
4. 调 `deleteObjects` 删除。

说明：这是失败补偿接口，不参与正常读写链路。

## 七、上传与业务写入流程

### 7.1 头像上传

1. 客户端持有 `sessionGrant`。
2. 调 `getQiniuUploadToken({ type: 'avatar', sessionGrant, ext })`。
3. 直传七牛。
4. 把返回的 `ref` 传给 `login/updateProfile` 入库。

### 7.2 创建宠物

1. 进入页面先调 `reservePetSpaceId`。
2. 客户端缓存 `petSpaceId + petUploadGrant`。
3. 每次上传封面/相册图时调用 `getQiniuUploadToken`。
4. 表单提交时把 `petSpaceId + avatarRef + coverRef` 一起传给 `createPetSpace`。

### 7.3 新增记忆

1. 进入页面先调 `getPetUploadGrant(petSpaceId)`。
2. 多图上传全部复用同一个 `petUploadGrant`。
3. 提交 `mediaRefs` 给 `addMemory`。

---

## 八、失败补偿

性能优先前提下，补偿采用“客户端立即清理 + 服务端轻量删除接口”，不引入上传日志表，不引入上传回调。

### 8.1 必做：`cleanupRefs`

新增轻量云函数：

```js
{
  refs: Ref[]
}
```

校验逻辑：

1. 必须有微信登录态。
2. 从 `sessionGrant` 拿到 `uid`。
3. 只允许删除 key 前缀匹配 `project/star-pet/uploads/users/{uid}/` 的对象。

使用时机：

1. `createPetSpace` / `addMemory` / `updateProfile` 提交失败时，客户端 best-effort 调一次。
2. 页面放弃编辑时，清理由本次会话新上传但未提交的 ref。

### 8.2 必做：本地待提交列表

前端页面维护：

```js
pendingUploadedRefs: Ref[]
```

规则：

1. 上传成功后加入列表。
2. 业务提交成功后，从列表移除已入库 ref。
3. 提交失败或用户放弃时，把列表传给 `cleanupRefs`。

这样不会增加服务端热路径写入，也能覆盖大多数孤儿图。

### 8.3 可选：定时保底清理

如果后续观察到孤儿文件仍明显增长，再加定时任务：

1. 仅扫描最近 N 天上传前缀。
2. 只处理明显不再被任何表引用的 key。

这项不是当前版本必做，避免引入大范围 DB 全表扫描。

## 九、删除与替换策略

保留“先写 DB，后删旧文件”。

补充约束：

1. `deleteObjects` 必须返回删除结果摘要：成功数、失败数、失败 key。
2. 主流程可以吞删除异常，但必须 `console.error` 输出失败明细。
3. 同 key 去重后再删。
4. 单批 500 个对象，避免超七牛 batch 限制。

建议返回结构：

```js
{
  successCount: 3,
  failCount: 1,
  failedKeys: ['project/...']
}
```

### 9.1 用户头像替换必须删除旧文件

这个问题必须明确纳入主流程。否则用户每改一次头像，都会在 `users/{uid}/avatars/` 下留下一个新文件，最终形成稳定增长的孤儿对象。

处理原则：

1. 只要 `users.avatarRef.key` 发生变化，就必须尝试删除旧 key。
2. 删除动作发生在 users 文档更新成功之后。
3. 删除失败不回滚主流程，但必须记录日志，交给后续兜底清理。
4. 如果新旧 key 相同，则不删。

推荐写法：

1. `login` 或 `updateProfile` 先读当前用户的 `avatarRef`。
2. 写入新的 `avatarRef`。
3. 比较 `oldAvatarRef.key !== newAvatarRef.key`。
4. 不同则异步调用 `deleteObjects([oldAvatarRef])`。

说明：

1. 这一步只涉及单个旧文件删除，成本极低，不会影响性能目标。
2. 头像是高频替换资源，这条规则必须作为验收项，而不是可选优化。

### 9.2 宠物头像、封面替换同样适用

`pet_spaces.avatarRef`、`pet_spaces.coverRef` 与用户头像规则一致：

1. 先更新 DB。
2. 识别被替换掉的旧 ref。
3. 去重后删除旧 key。

`updateMemory` 则继续按 `oldRefs - newRefs` 求差集删除。

## 十、`_shared/storage` 方案结论

保留 `cloudfunctions/_shared/storage.js` 和 `cloudfunctions/_shared/storage.lite.js`，通过 `scripts/sync-shared.js` 物理同步到各云函数目录。这是当前工程结构下最务实的方案。

### 10.1 为什么保留

1. 微信云函数独立打包，不能稳定依赖工作区共享模块。
2. 物理复制比手工拷贝可靠，且部署结果可预期。
3. 读侧和写侧拆成 `lite/full` 两个版本，可以避免所有读侧函数都带上 `qiniu` SDK。

### 10.2 边界

`_shared/storage` 只允许放三类能力：

1. URL 拼接：`buildUrl(ref)`、`assetUrl(path)`
2. 七牛删除：`deleteObjects(refs)`
3. 存储常量：`BUCKET`、`CDN_HOST`、`KEY_PREFIX`

明确不放：

1. grant 签发和验签
2. 业务权限判断
3. `petSpaceId` 归属校验
4. 表字段转换和业务拼装

这些内容分别放到 `_shared/grant.js` 或各自云函数内部，避免 `_shared/storage` 膨胀成大杂烩。

### 10.3 约束

1. `_shared/storage.js` 是唯一源文件，禁止直接修改各云函数下的复制产物。
2. 每次修改 `_shared/*` 后必须运行 `node scripts/sync-shared.js`。
3. CI 必须跑 `node scripts/sync-shared.js --check`，防止漂移。
4. `deleteObjects` 必须返回删除摘要，并在失败时打印日志，不能静默吞错。

### 10.4 结论

这个方案不是“优雅共享”，但在当前平台限制下是正确折中。它足够轻，也足够可控，前提是共享文件保持小、同步检查自动化。

## 十一、安全边界

1. 七牛 `SecretKey` 只在云函数环境变量。
2. `sessionGrant`、`petUploadGrant` 都必须带版本号和过期时间。
3. HMAC 密钥与七牛密钥分离，单独使用 `UPLOAD_GRANT_SECRET`。
4. `getQiniuUploadToken` 只允许签发精确到 `bucket:key` 的 token，不允许前缀级别 scope。
5. `ext` 和 MIME 只允许白名单：`jpg/jpeg/png/webp`。
6. 上传大小限制建议 8MB，不再给 10MB，减少移动端上行耗时和存储浪费。
7. 如需恢复图片审核，应放在“业务写入前”而不是“展示前”。

## 十二、性能评估

### 11.1 热路径

单次上传签 token 的执行内容：

1. 读取微信上下文。
2. 验两个短字符串签名。
3. 生成 key。
4. 调七牛 SDK 本地签 token。

特点：

1. 无 DB。
2. 无外部 HTTP。
3. CPU 开销极低。

### 11.2 冷路径

会查库的地方只剩：

1. `login` 首次创建/补齐 uid。
2. `reservePetSpaceId` 创建前置步骤。
3. `getPetUploadGrant` 页面进入时校验归属。

这几个调用频率明显低于图片上传频率，符合性能优先目标。

## 十三、实施清单

### 12.1 第一阶段：共享能力

1. 新增 `_shared/grant.js`
   - 提供 `signGrant(payload)`、`verifyGrant(token)`。
   - 使用 `UPLOAD_GRANT_SECRET` 做 HMAC-SHA256。
2. 统一 `_shared/storage.js`
   - 提供 `buildUrl(ref)`。
   - 提供 `deleteObjects(refs)`。
   - 返回删除摘要并打印失败日志。
3. 补环境变量
   - `QINIU_ACCESS_KEY`
   - `QINIU_SECRET_KEY`
   - `QINIU_CDN_HOST`
   - `UPLOAD_GRANT_SECRET`

验收：

1. 本地能完成 grant 签发与验签单测。
2. `deleteObjects` 对不存在 key、重复 key、空数组都可安全执行。

### 12.2 第二阶段：登录态与授权

1. 改 `login`
   - 新用户生成 `uid`。
   - 返回 `sessionGrant`。
2. 改 `reservePetSpaceId`
   - 返回 `petSpaceId + petUploadGrant`。
3. 新增 `getPetUploadGrant`
   - 仅在进入编辑宠物/新增记忆页时调用一次。

验收：

1. 同一登录态下，多次上传不再额外查 `users`。
2. 篡改 `petSpaceId` 或 grant 内容时，`getQiniuUploadToken` 必须拒绝。

### 12.3 第三阶段：上传链路

1. 改 `getQiniuUploadToken`
   - 不再查 `users`。
   - avatar 仅依赖 `sessionGrant`。
   - 宠物相关图片依赖 `sessionGrant + petUploadGrant`。
   - 返回 `ref + url`。
2. 改 `miniprogram/utils/storage/qiniu.js`
   - 调用云函数时透传 grant。
   - 上传成功后直接返回 `{ ref, url }`。
3. 改 `components/image-uploader/index.js`
   - 上传成功后返回 `{ ref, url }`。
   - 每次上传成功后，把 ref 加入页面级 `pendingUploadedRefs`。

验收：

1. 连续上传 5 张记忆图片时，只出现一次页面初始化查库。
2. 上传热路径日志里不应出现任何 DB 查询。

### 12.4 第四阶段：业务写入与旧文件删除

1. 改 `login` 或 `updateProfile`
   - 入库 `avatarRef`。
   - 如果旧头像 key 与新头像 key 不同，删除旧头像文件。
2. 改 `createPetSpace`
   - 入库 `avatarRef / coverRef`。
3. 改 `updatePetSpace`
   - 对替换掉的 `avatarRef / coverRef` 做去重删除。
4. 改 `addMemory`
   - 入库 `mediaRefs`。
5. 改 `updateMemory`
   - 继续按差集删除被移除图片。

验收：

1. 用户连续改 3 次头像后，七牛仅保留最后一次头像文件。
2. 宠物封面替换后，旧封面文件被删除。
3. 记忆编辑移除图片后，被移除对象被删除。

### 12.5 第五阶段：失败补偿

1. 新增 `cleanupRefs`
   - 只允许删除当前 uid 前缀下的 key。
2. 页面在以下时机触发回收：
   - 提交失败
   - 主动取消编辑
   - 上传后又被本地替换掉但未提交
3. 页面提交成功后，从 `pendingUploadedRefs` 中清掉已落库 ref。

验收：

1. 上传成功但 `addMemory` 失败时，客户端会发起回收。
2. 头像上传后未保存就离开页面时，新头像对象会被回收。

### 12.6 第六阶段：读侧兼容

1. 改所有读侧云函数
   - 从 ref 拼 `avatarUrl / coverUrl / mediaUrls`。
2. 不改 wxml 绑定字段。

验收：

1. 页面无需改模板即可正常展示。
2. DB 中确认不存在 url 冗余字段。

### 12.7 第七阶段：回归测试

1. 新用户注册并上传头像。
2. 连续替换头像 3 次。
3. 创建宠物并上传头像/封面。
4. 编辑宠物并替换封面。
5. 新增记忆并上传多张图片。
6. 编辑记忆并删除部分图片。
7. 模拟提交失败，验证 `cleanupRefs` 生效。

最终验收标准：

1. 上传热路径无 DB 查询。
2. 用户头像、宠物头像、宠物封面替换后，旧文件不会持续堆积。
3. 业务提交失败时，新增图片大部分可在当前会话内被回收。
4. 读侧接口继续返回前端所需的 url 字段。

---

## 十四、不做的事

1. 不做上传日志表。
2. 不做多 provider 运行时分发。
3. 不做上传后回调入库。
4. 不做对象重命名/搬运。
5. 不做任何旧字段兼容层。
6. 不做全量孤儿文件扫描任务作为当前版本依赖。

这些都能做，但不符合“性能第一、实现不能太重”的约束。

---

## 十五、关键决策

1. 去掉每次上传都查 `users` 取 `uid`。
2. 去掉“只要传 `petSpaceId` 就能拿到对应路径 token”的设计。
3. 把失败补偿定义为正式链路的一部分，而不是待办。
4. 用户头像、宠物头像、宠物封面替换后，旧文件必须删除。
5. 保留 `_shared/storage` 同步机制，但限制其职责边界。
6. 不保留任何旧字段、旧接口、旧上传逻辑。

## 十六、推荐默认参数

```js
{
  qiniu: {
    bucket: 'cl8023',
    uploadUrl: 'https://upload-z2.qiniup.com',
    cdnHost: 'https://qiniu.cdn.cl8023.com',
    keyPrefix: 'project/star-pet/uploads'
  },
  grant: {
    sessionTtlMs: 24 * 60 * 60 * 1000,
    petUploadTtlMs: 20 * 60 * 1000
  },
  upload: {
    maxBytes: 8 * 1024 * 1024,
    allowedExts: ['jpg', 'jpeg', 'png', 'webp']
  }
}
```
