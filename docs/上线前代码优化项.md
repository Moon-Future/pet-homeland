# 上线前代码优化项

> 记录日期：2026-06-10
> 范围：本清单不包含内容安全自动审核；首版内容治理先以人工审核链路兜底。

## P0 必须处理

### 1. 七牛对象 Ref 归属校验

当前 `avatarRef`、`coverRef`、`mediaRefs` 写入服务端时只校验字段格式，没有校验对象 key 是否属于当前用户和当前小窝。由于更新头像、封面、回忆时会用服务端七牛密钥删除旧对象，如果用户提交了不属于自己的 key，可能导致越权引用或误删他人文件。

涉及位置：

* `cloudfunctions/login/index.js`
* `cloudfunctions/createPetSpace/index.js`
* `cloudfunctions/updatePetSpace/index.js`
* `cloudfunctions/addMemory/index.js`
* `cloudfunctions/updateMemory/index.js`

优化要求：

* 服务端保存 Ref 前，强制校验 `storage === 'qiniu'`、`bucket` 为项目桶。
* 头像 key 必须匹配当前用户前缀：`project/star-pet/uploads/users/{uid}/avatars/`。
* 小窝封面、相册、回忆图片 key 必须匹配当前用户和当前小窝前缀：`project/star-pet/uploads/users/{uid}/pet-spaces/{petSpaceId}/...`。
* 删除旧对象前再次校验 key 属于当前用户前缀，避免服务端管理密钥误删非本人对象。
* `createPetSpace` 使用预分配 `_id` 时，必须校验 `_id` 与 `petUploadGrant.petSpaceId` 一致，不能只信任客户端传入的 `_id`。

验收标准：

* 伪造其他用户或其他小窝的 Ref 时，云函数返回失败，不写入 DB。
* 更新头像、封面、回忆时，只会删除当前用户前缀下的对象。
* 已有正常上传流程不受影响。

落地记录：

* 已新增共享校验模块 `cloudfunctions/_shared/upload-ref.js`，并同步到 `login`、`createPetSpace`、`updatePetSpace`、`addMemory`、`updateMemory`。
* `createPetSpace` 已校验 `petUploadGrant` 与预分配 `_id` 一致；正式环境需要为 `createPetSpace` 配置 `UPLOAD_GRANT_SECRET`。

## P1 上线前建议处理

### 2. 互动接口补齐公开审核状态校验

`getPetSpaceDetail` 已经要求访客访问 `discover` 小窝时必须 `reviewStatus === 'approved'`，但互动相关云函数只判断 `visibility`，知道小窝 ID 的用户可能对待审或未通过公开审核的小窝互动。

涉及位置：

* `cloudfunctions/interactPetSpace/index.js`
* `cloudfunctions/getPetInteractionSummary/index.js`

优化要求：

* 非主人访问 `discover` 小窝时，必须同时满足 `status === 'active'` 和 `reviewStatus === 'approved'`。
* `share` 小窝保持分享链接可访问、可轻互动的逻辑。
* 被管理员隐藏、拒绝或待审的小窝不能被访客互动，也不能读取访客互动摘要。

验收标准：

* `discover + pending_review/rejected/hidden`：非主人互动失败。
* `discover + approved`：非主人互动正常。
* `share`：非主人通过分享入口互动正常。

落地记录：

* `interactPetSpace` 和 `getPetInteractionSummary` 已补齐访客访问判断：`discover` 必须为 `approved`，`share` 保持可访问。

### 3. 关键统计与额度更新改为事务或可重建

新增、编辑、删除回忆时，当前流程会分步写 `memories`、`media`、`pet_spaces.stats`、`users.stats`。任一步失败都可能导致统计和真实数据不一致。图片额度也是先 `count()` 再写入，并发请求可能突破每人 30 张限制。

涉及位置：

* `cloudfunctions/addMemory/index.js`
* `cloudfunctions/updateMemory/index.js`
* `cloudfunctions/createPetSpace/index.js`

优化要求：

* 优先将回忆写入、媒体元信息写入、统计增减放进数据库事务。
* 如果事务实现成本较高，至少把展示统计统一改为实时聚合，并提供管理员修复云函数重算 `users.stats`、`pet_spaces.stats`。
* 图片额度检查需要和写入绑定，避免并发提交同时通过额度校验。

验收标准：

* 模拟 `media` 写入失败时，不会留下半成功的回忆和错误统计。
* 并发提交多条回忆时，图片总数不会超过额度。
* 删除回忆后，统计和媒体状态能保持一致。

落地记录：

* 已新增管理员云函数 `rebuildStats`，可按真实 `pet_spaces`、`memories` 重算 `users.stats` 和 `pet_spaces.stats`。
* 写入端仍使用现有 `_.inc` 增量维护；如后续需要强并发额度锁，再引入事务或独立额度计数集合。

## P2 数据增长后优化

### 4. 小窝列表与详情统计查询降本

当前 `getMyPetSpaces` 会对每个小窝分页扫描 `memories` 计算精确统计，`getPetSpaceDetail` 也会扫描所有可见回忆计算 `memoryCount` 和 `mediaCount`。用户数据变多后，详情和列表接口会越来越慢。

涉及位置：

* `cloudfunctions/getMyPetSpaces/index.js`
* `cloudfunctions/getPetSpaceDetail/index.js`
* `cloudfunctions/getMemories/index.js`

优化方向：

* 主展示优先使用已维护的 `stats` 字段。
* 保留后台重算任务，用于修正历史不一致。
* 需要精确统计时按需加载，不阻塞首屏详情展示。

落地记录：

* `getMyPetSpaces` 已改为直接使用 `stats`，不再逐个小窝扫描 `memories`。
* `getPetSpaceDetail` 在主人/管理员视角直接使用 `stats`；访客视角仍按可见回忆计算，避免泄露待审内容数量。

### 5. 互动访客统计避免全量扫描

`getPetInteractionSummary` 主人视角会扫描全部历史 `interactions` 去重访客，互动数据增长后会变慢。

涉及位置：

* `cloudfunctions/getPetInteractionSummary/index.js`

优化方向：

* 在 `pet_spaces.stats` 增加可维护的访客累计字段，或新增按日/按小窝聚合表。
* 主人详情页优先读取聚合结果，历史精确统计改为后台任务。

落地记录：

* `interactPetSpace` 已在访客首次互动时维护 `stats.visitorCountAllTime`。
* `getPetInteractionSummary` 已优先读取 `stats.visitorCountAllTime`，不再扫描全部历史互动去重。

### 6. 星空广场分页与随机策略优化

当前 `getDiscoverPetSpaces` 固定取最近 100 条后随机截取，数据量变大后老内容曝光不足，也不支持稳定分页。

涉及位置：

* `cloudfunctions/getDiscoverPetSpaces/index.js`

优化方向：

* 增加分页游标，避免每次只看最近 100 条。
* 可增加 `discoverWeight`、`lastFeaturedAt` 或随机桶字段，用于更稳定的广场曝光策略。

落地记录：

* `getDiscoverPetSpaces` 已支持可选 `cursor`，返回 `nextCursor`；默认不传游标时保持最近 100 条随机的旧行为。

## 上线配置检查

### 7. 数据库权限与索引

上线前需要在微信云开发控制台确认集合权限和索引，而不是只依赖代码约束。

权限要求：

* `users`、`pet_spaces`、`memories`、`media`、`interactions`、`feedbacks`、`reports` 均设置为“所有用户不可读写”。
* 小程序端不直接读写业务集合，统一走云函数。

建议索引：

* `users.openid` 唯一索引。
* `users.uid` 唯一索引。
* `pet_spaces.identityNo` 唯一索引。
* `interactions(petSpaceId, openid, type, dateKey)` 唯一或事务锁。
* `reports(reporterOpenid, targetType, targetId, status)` 普通组合索引，用于防重复举报查询。
